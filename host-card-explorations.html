01
Direction one

The datasheet

Stacked rails — Agent · Protocol · Apps — read top-to-bottom like a hardware datasheet. Every field has a fixed label column, a monospaced value, and an optional qualifier tag. Off capabilities stay in the list (strike-through), because absence is informative. A warm hairline marks anything that drifted from the previous host.

02
Direction two

The capability matrix

A single tight table — every host becomes the same row layout, so you can scan three of them in a row and immediately see who supports what. Identity on top; protocol in three balanced cells; then two stacked matrices — base MCP client capabilities (roots · sampling · elicitation · tasks · experimental) and the Apps extension caps — under labeled bands. Modeled after kubectl get pods output, but typeset.

What's working: Three hosts → three identical layouts → eye snaps to the cells that differ. The M / + gutter borrows from git, which developers already read fluently. Stacking the client capabilities matrix on top of the Apps extension matrix lets one card answer two questions at once: "does this host speak the base protocol I need?" and "does it speak Apps UI?" — both with the same scanning vocabulary. Trade-off: the visual density is high; you trade canvas-poetry for ledger-precision. Best paired with a separate detail pane on hover.
03
Direction three

The inspector tree

The host as an expandable property tree, modeled on a code editor's outline view. Group headers act as sections; every leaf has a name, an inline value, and a one-character status gutter on the left — same vocabulary as git status. Right at home in a dev tool.

What's working: Information density rivals the canvas, but reading is left-to-right, top-to-bottom. The git-style gutter is a small thing that pays off enormously — eyes track to M markers reflexively. Trade-off: doesn't dramatize "host shape" the way the canvas does. 04
Direction four

The character card

A bolder swing — each host gets a small portrait. Provider glyph, host name in serif italic, model in mono, plus a strip of stat-block fields and an LED-panel capability strip below. Designed for the moment when you have six hosts and need to remember which is which. Personality on purpose.

What's working: Two hosts a row, each instantly recognizable from across a screen — provider glyph + serif name + signature brand stripe. The "NEW" pip on a freshly-acquired capability is the cleanest expression of the morph idea. Trade-off: the most visually expressive option, but also the heaviest. Probably the best fit for the host picker, with one of the lighter directions inside the builder canvas.
A few cross-cutting decisions worth calling out:
Off ≠ hidden. Every direction keeps unsupported capabilities visible (strike-through, dashed cell, gutter −). The shape of "what's missing" is the most useful answer mcpjam can give a developer comparing hosts.
Diff is left-aligned. The eye picks up a thin vertical bar (datasheet), a single-letter gutter (matrix, inspector), or a small NEW pip (card) much faster than a colour-fill across an entire row. The animation can flash and fade — the marker stays.
Type does the heavy lifting. Fraunces for identity (lets a Claude card and a Cursor card feel different at a glance) · Geist for labels · JetBrains Mono for values. No icons in the data — they're not needed and they add noise.

Recommended pairing → Direction 03 (inspector tree) as the canvas focal view in the builder; Direction 02 (matrix) when comparing two hosts side-by-side in a diff overlay; Direction 04 (card) for the host picker grid.
