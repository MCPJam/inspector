What it's good at: spotting a host's weakness in <2 seconds. Block of red in a column = host has a gap. The "spread" column flags rows where hosts disagree — the most interesting rows for a spec author.

Where it gives up ground: raw pass/total takes a second look. Heavy colour can feel scoreboard-y in a tool that's mostly for diagnosis. Doesn't survive print/screenshots as gracefully.

What it's good at: easy to read a partial pass (4/6 looks meaningfully different from 0/6 at a glance, where a heatmap can blur "0%" and "low %"). Donuts in headers give an instant per-host verdict.

Where it gives up ground: taller rows — fewer capabilities visible per scroll. Meters can read as progress bars rather than test outcomes, which is a slight semantic mismatch.

V1

Ledger

Closest to today's cross-host-matrix.tsx. Lowest engineering lift — same table primitives, add tree expand + capability rollups.

Recommended starting point.

V2

Heatmap

Best for the "scoreboard" framing — if this is going to be a marketing surface or public conformance page, lean here. The "spread" column is genuinely novel.

V3

Scorecard

Best when partial passes matter (4/6 vs 0/6). Donut headers give per-host verdicts free. Slightly heavier; consider for the dedicated "Conformance" tab rather than the in-suite view.

Sketches use design-system tokens (background, primary, success/warning/destructive, muted, JetBrains Mono). Fraunces is loaded from Google Fonts for the editorial serif accents; production swap to the DS font stack is one variable change.
