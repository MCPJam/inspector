`MCPClientManager` is the piece that opens and manages MCP server connections.

You tell it where a server lives, connect once, then use it to:

- list tools
- call tools
- read resources
- work with prompts
- manage more than one server at a time

## When to Use MCPClientManager

Use `MCPClientManager` when you need to:
- Connect to one or more MCP servers
- Execute tools programmatically (without an LLM)
- Build applications that aggregate tools from multiple servers
- Set up test environments for your MCP server

## Two Ways to Connect

Most of the time, you will connect in one of these two ways:

### Local server: STDIO

Use this when the server should run on your machine. You give the SDK a command, and it starts the server for you.

```typescript
import { MCPClientManager } from "@mcpjam/sdk";

const manager = new MCPClientManager({
  myServer: {
    command: "node",
    args: ["./my-server.js"],
    env: {
      API_KEY: process.env.MY_API_KEY,
    },
  },
});
```

Use STDIO for:
- Local development
- Servers you're building
- CLI-based MCP servers
- Servers requiring local resources

### Remote server: HTTP

Use this when the server already has a URL and runs somewhere else.

```typescript
const manager = new MCPClientManager({
  asana: {
    url: "https://mcp.asana.com/sse",
    requestInit: {
      headers: {
        Authorization: `Bearer ${process.env.ASANA_TOKEN}`,
      },
    },
  },
});
```

Use HTTP for:
- Production MCP servers
- Third-party integrations (Asana, Slack, etc.)
- Shared team servers
- Serverless deployments

### HTTP Authentication Options

For remote servers, there are three common auth setups:

- `accessToken` if you already have a bearer token
- `refreshToken` + `clientId` if you want the SDK to fetch and refresh access tokens
- `authProvider` if you need a custom auth flow

If you already have a refresh token, the SDK can handle token exchange and refresh for you:

```typescript
const manager = new MCPClientManager({
  asana: {
    url: "https://mcp.asana.com/mcp",
    refreshToken: process.env.ASANA_REFRESH_TOKEN!,
    clientId: process.env.ASANA_CLIENT_ID!,
    clientSecret: process.env.ASANA_CLIENT_SECRET,
  },
});
```

With this setup, the SDK:

- exchanges the refresh token for an access token during connection
- automatically retries with a fresh access token when the server returns an auth challenge
- stores rotated refresh tokens returned by the authorization server

<Note>
`refreshToken` is only supported for HTTP servers. When you use it, do not also set `accessToken`, `authProvider`, or an `Authorization` header in `requestInit.headers`.
</Note>

## Basic Flow

After you define a server, you still need to connect to it:

```typescript
// Connect to a specific server
await manager.connectToServer("myServer");

// Now you can use it
const tools = await manager.listTools("myServer");
console.log("Available tools:", tools.tools.map((tool) => tool.name));

// Always disconnect when done
await manager.disconnectServer("myServer");
```

<Note>
Disconnect when you're done, especially in tests, so subprocesses and network connections get cleaned up.
</Note>

## Working with Multiple Servers

One manager can handle multiple servers at once:

```typescript
const manager = new MCPClientManager({
  math: {
    command: "npx",
    args: ["-y", "@modelcontextprotocol/server-everything"],
  },
  asana: {
    url: "https://mcp.asana.com/sse",
    requestInit: {
      headers: { Authorization: `Bearer ${token}` },
    },
  },
  github: {
    url: "https://mcp.github.com/sse",
    requestInit: {
      headers: { Authorization: `Bearer ${ghToken}` },
    },
  },
});

// Connect to all of them
await Promise.all([
  manager.connectToServer("math"),
  manager.connectToServer("asana"),
  manager.connectToServer("github"),
]);

// Get tools from every connected server
const allTools = await manager.getTools();
// Now allTools contains tools from math, asana, and github
```

## Calling Tools Directly

You do not need an LLM to test a tool. This is handy for unit tests:

```typescript
// Direct tool execution
const result = await manager.executeTool("math", "add", { a: 5, b: 3 });
console.log(result); // 8

// Test your server's tools deterministically
expect(result).toBe(8);
```

## Integration with TestAgent

A common pattern is:

1. connect your servers
2. get their tools
3. pass those tools to `TestAgent`

```typescript
const manager = new MCPClientManager({
  myServer: { command: "node", args: ["./server.js"] },
});

await manager.connectToServer("myServer");

// Get tools for TestAgent
const tools = await manager.getTools();

// Create agent with those tools
const agent = new TestAgent({
  tools,
  model: "anthropic/claude-sonnet-4-20250514",
  apiKey: process.env.ANTHROPIC_API_KEY,
});

// Now ask the agent something
const result = await agent.prompt("Add 2 and 3");
```

## Error Handling

Connections can fail, so use `try/catch` in real apps:

```typescript
try {
  await manager.connectToServer("myServer");
} catch (error) {
  console.error("Failed to connect:", error.message);
  // Handle gracefully - maybe fall back to another server
}
```

## Health Checks

If you want to check that a server is still alive:

```typescript
try {
  manager.pingServer("myServer");
} catch {
  console.warn("Server not responding, reconnecting...");
  await manager.disconnectServer("myServer");
  await manager.connectToServer("myServer");
}
```
