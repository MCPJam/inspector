## How MCP E2E Testing Works

E2E testing simulates real user workflows by testing complete chains of interactions. For MCP servers, this means testing how they work when used by actual LLMs and agents in real-world scenarios.

### Why Test MCP Servers Differently?

- **APIs** are consumed by other APIs or web clients
- **MCP servers** are consumed by LLMs and agents (like Claude Desktop, Cursor)
- We need to simulate the actual user environment where MCP servers operate

### How It Works

1. **Setup**: Connect your MCP server to a testing agent
2. **Simulate**: Configure the agent to behave like a real user (e.g., Claude Desktop)
3. **Test**: Have the agent run realistic user queries
4. **Verify**: Check the agent's trace to confirm correct tool usage

### Example: PayPal MCP Test

```
1. Connect PayPal MCP server to testing agent
2. Ask: "Create a refund for order ID 412"
3. Agent runs the query using MCP tools
4. Verify: Did it call create_refund tool successfully?
```

An LLM judge can analyze the agent's trace to determine if the test passed.

If you run evals through the SDK, pass the connected `MCPClientManager` into [`TestAgent`](/sdk/reference/test-agent) (or pass `agent` / `mcpClientManager` to reporting APIs) so MCPJam can **persist replay credentials** for authenticated HTTP servers **and** capture **MCP App widget HTML** (via `readResource` on `ui.resourceUri`) for **offline widget replay** in Eval traces. Without `mcpClientManager` on `TestAgent`, uploaded traces still include messages and spans but not embedded app previews.

### Pass/fail and tool errors

Suite runs in the Inspector combine **expected vs. actual tool matching** with **tool execution outcome**: if a tool returns an MCP error (`isError`) or the trace records a failed tool step, the iteration is marked **failed** by default, even when the right tools were invoked. That matches the behavior documented for the SDK in [Tool execution and `passed`](/sdk/reference/eval-reporting#tool-execution-and-passed).

To only score structural matching for a test case, set **`failOnToolError: false`** on that case’s **advanced config** (extra fields are accepted on the run payload). SDK uploads use the same flag on [`MCPJamReportingConfig.failOnToolError`](/sdk/reference/eval-reporting#mcpjamreportingconfig).
